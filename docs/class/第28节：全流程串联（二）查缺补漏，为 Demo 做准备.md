# 全流程串联（二）：查缺补漏，为 Demo 做准备

上一节把人推主干拉通了：一句话进来，八站走完，账全对得上。这节接着补另外半边——**钟推链路**（定时触发 → ReAct → notify 推送）、**跨重启恢复**、**多 Agent 并存**，再做一轮稳定性打磨，最后列出两个 Demo 上场前的前置条件清单。做完这节，底座就算真正"跑得稳"了。

---

## 一、这一步要干什么

一句话：**上一节验证的是"有人问、答得对"，这一节验证的是"没人问、也干活；重启了、不失忆；多个 Agent、互不打架"。**

这三件事正好是 Agent OS 区别于"一个 Agent Demo"的地方：Demo 只要在演示的那几分钟里活着就行，底座要在没人看着的时候也持续正确。

## 二、把钟推链路拉通

钟推链路 = 人推链路换了个触发头（AgentScheduler 代替人）、多了个推送尾（notify 代替屏幕输出），中间完全复用：

![钟推链路五站：定时触发、统一入口、ReAct、notify 推送、落库](../../website/public/images/class-28-1.svg)

还是用上一节的对账法。配一个测试 Profile：`schedules` 设成每两分钟触发、消息是"查一下北京天气，把穿搭建议推送出去"，`notify_channels` 指向一个测试 webhook。等它自己跑一轮，然后对账——这次触发应该留下且只留下：

- `sessions` 表：channel 和 user 都是 `scheduler` 的那条 Session **被复用**（第二次触发不新建，历史追加在同一条里，25 节定的规则）；
- `llm_calls`：两条（要调工具那轮 + 组织答复那轮）；
- `tool_invocations`：两条——`http_get`（查天气）和 `notify`（推送），都 `success=true`；
- webhook 那头真的收到了消息。

**重点核对两处交接**：一是 Session 复用——连续两次触发之后 `sessions` 表里还是一条记录、`messages_json` 变长了，而不是冒出两条；二是 notify 的失败路径——把 webhook 地址故意改成白名单外的域名，确认 Sandbox 拦下、`tool_invocations` 里有 `success=false` 的记录、**调度器本身没死**（下一个触发点照常来）。

## 三、跨重启恢复

底座的状态全部外置（SQLite + 文件系统），进程本身无状态——这句架构承诺，重启一次就能验证：

1. 跑几轮对话、攒几条记忆、让定时任务触发过至少一次，然后 `kill` 掉进程；
2. 重新 `oryxos serve`，逐项核对：
   - **Session 回来了**：`GET /sessions/{id}` 能查到重启前的完整历史（SQLite 里本来就在）；
   - **记忆回来了**：新对话里 Agent 还记得核心记忆的内容（`MEMORY.md` 是文件，天然跨重启）；
   - **定时任务回来了**：启动日志里能看到 `schedules` 重新注册，到点照常触发——`AgentScheduler` 是启动时扫 Profile 重建的，不依赖任何进程内残留；
   - **审计不断档**：重启前后的 `llm_calls` 记录连续，没有丢一段。

哪一项没回来，说明有状态偷偷留在了进程内存里——这是走向分布式前必须掐灭的隐患（无状态实例是技术方案定下的前提）。

## 四、多 Agent 并存

"OS"在核心阶段的最小体现：多个 Profile 在同一实例上同时可用。配两个差异明显的 Profile（比如一个只有文件工具、一个只有 HTTP 工具），交替使用，核对三条隔离边界：

- **工具隔离**：Profile A 的对话里，模型拿到的工具列表里没有 B 独有的工具（`tools` 字段过滤生效）；
- **会话隔离**：A 和 B 各自的 Session 互不串——`session_id` 里带着 profile，天然分开，但要防止哪处代码用了"默认 Profile"把两边混了；
- **定时隔离**：A 的定时任务抛异常，B 的定时任务下一个触发点照常执行（25 节的失败隔离，在多 Agent 场景下再验一次）。

## 五、把"能跑"打磨成"跑得稳"

最后一轮打磨，几件小事，件件影响 Demo 现场的体感：

- **超时兜底**：LLM 调用、工具执行、notify 推送都要有超时，一处卡死不能拖住整个虚拟线程；Agent 调用整体 60 秒上限（26 节定的）。
- **错误信息可读**：`error_message` 里存的是人能看懂的原因（"域名不在白名单内: evil.com"），不是一坨堆栈——审计表是给人查的。
- **日志一次触发一条主线**：一次对话/一次定时触发，从日志里能按 session_id 串出完整过程，不用在几百行里捞。
- **MCP server 失联不拖垮启动**：外部 MCP 连不上，记 WARN、跳过它的工具，OryxOS 自己照常起——外部依赖的可用性不能变成自己的可用性。

## 六、Demo 前置条件清单，做完怎么验

31 节两个 Demo 要用的环境，这节末尾一次配齐、逐项打勾：

![六项前置条件：白名单、通知渠道、新闻 MCP、记忆偏好、定时配置、跨重启](../../website/public/images/class-28-2.svg)

验收合并成一句话能说清的三条：

- **钟推全链路**：测试 Profile 到点自动跑完"查天气 → 推送"，对账不多不少；Session 复用、失败不崩调度器。
- **重启不失忆**：kill 再起，Session、记忆、定时、审计四样全部无损恢复。
- **多 Agent 不打架**：两个 Profile 并存，工具、会话、定时三条边界都验证过。

到这里，底座部分（第一部分）全部完工且跑得稳。但注意一个事实：我们到现在还没有真正"定义"过一个业务 Agent——测试用的 Profile 都是顺手配的。下一节讲这门课最关键的一次视角转换：**底座是底座，Agent 是 Agent**——用 Skill 把一个业务 Agent 声明式地定义出来。
