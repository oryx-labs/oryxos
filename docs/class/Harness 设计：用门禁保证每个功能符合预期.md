# Harness 设计：用门禁保证每个功能符合预期

这份文档沉淀"我们怎么给 AI 开发上门禁"的完整设计——为什么需要、按什么哲学分类、具体有哪几道、分别挂在流程的哪个位置。它是 `/oryxos-lesson-dev` skill 的设计依据，也是第 6 节"Harness 给边界"这句方法论在本项目里的完整落地样本。

配套阅读：`Spec-Kit 执行指导：从课件到代码（以第16节为例）.md`（流程）、各节课件的"验收 harness"部分（测试规格）、`docs/TechnicalSolution.md`（权威设计源）。

---

## 一、为什么需要门禁：AI 逐节开发的真实风险

执行交给 AI 之后，风险不是"写错"——写错测试会抓；真正的风险是**每节局部合理、整体漂移**，最后交付出来的不再是 `docs/TechnicalSolution.md` 里设计的那个 OryxOS。漂移有四种典型形态：

1. **发明**：造出文档里不存在的公共类、配置项、端点——单看都"挺合理"，累积起来系统长歪；
2. **顺手优化**：把已定的签名"改进"一下（`chat(sessionId, Profile, Prompt)` 少传一个参数）、把配置键换个"更好"的名字——下一节按文档喂料时全部对不上；
3. **提前实现**：把"有几样先别做"清单里的东西顺手做了（加个缓存、做个 fallback）——复杂度背上了，还没有信号支撑；
4. **臆造 API**：Spring AI 各 milestone 差异大，AI 最爱写出根本不存在的方法——编译期才炸算好的，更糟的是编出一个碰巧存在但语义不同的。

门禁就是针对这四种漂移设的关卡。我们交付的是 OryxOS 这个**最终整体**，不是十几段各自能跑的代码。

## 二、门禁哲学：一句话

> **能机器判的绝不留给人，机器判不了的绝不自行发挥。**

由此分出两类门禁：

| | 硬门禁 | 软门禁 |
|---|---|---|
| 谁来判 | 机器（测试、构建、静态检查、脚本核对） | 人（AI 停下报告，等确认放行） |
| 不过会怎样 | 构建失败 / 任务不得推进 | 流程暂停，等人拍板 |
| 管什么 | 结果可自动验证的事：测试绿不绿、不变量破没破、交付物在不在 | 需要判断的事：设计变更、文档冲突、范围调整 |
| 失败模式防的是 | 做错了没发现 | 自作主张 |

这跟课程一贯的"控制权在自己手里"是同一个原则：AI 可以执行一切，但**改变设计的权力永远不下放**。

## 三、七道门禁（H0～H6）

### H0 开工纪律（软性前置）

每节动手前必读三样：当节课件全文、`TechnicalSolution.md` 对应章节、前序各节的"本节交付物"清单。读完先做**依赖存在性检查**（做 17 节先确认 `ProviderService` 在），缺了停下——先补前序，不许跳节自造。

### H1 设计保真（软门禁，最高优先级）

- `docs/TechnicalSolution.md` 是唯一权威设计源。写下的每个公共概念必须能在技术方案或当节课件找到出处；两者冲突时**停下报告，不得自行仲裁**。
- 只创建"本节交付物"点名的公共类/接口/表/配置/端点。私有辅助类自由，但任何新的**对外概念**（public 类型、配置键、数据表、REST 路径、Profile 字段）都是设计变更：停下报告。
- 已定字面量逐字保真：`chat(sessionId, Profile, Prompt)`、`Sandbox.enforce(SandboxAction)`、`getOrCreate(channel, user, profileName)`、`file.allowed_paths` 这类签名和配置键，不做任何"顺手改进"。
- "有几样先别做"是**禁止实现清单**：不写其功能、不留 TODO 桩、不为它预埋参数。

### H2 跨节契约（软门禁 + 硬门禁各一半）

- 前序节交付的公共接口只可使用、不可修改——除非当节课件明确列为"改造点"（如 24 节四个 Tool 的 `execute` 首行）。要改：停下报告（软）。
- 本节结束时，**前序所有节的 harness 测试必须仍然全绿**（硬）——任何一个变红即破坏契约。
- 发现需要前序未交付的能力，停下报告，不得就地自造替代品（软）。

### H3 API 真实性（软门禁）

- 所有第三方 API（Spring AI / Spring / Picocli）编写前先在本地依赖里核实存在（`mvn dependency:tree`、翻 jar 源码）；核实不了的一律不写，停下报告。
- 不新增 plan 未列明的第三方依赖；确需新增：停下报告。

### H4 全局不变量（硬门禁，每节完成前逐条自查）

无论实现哪一节，以下六条永远成立，任一破坏即未完成：

1. 一切涉外 IO 执行首行过 `Sandbox.enforce`（Sandbox 就位前的节：留接口调用位，注明 24 节接线）；
2. 每次 LLM 调用成败都落 `llm_calls`，每次工具执行成败都落 `tool_invocations`；
3. 凭证只走环境变量，grep 无明文 key；
4. `session_id` 只在 `SessionManager` 内拼接；
5. 全程同步阻塞——出现 Reactor / `CompletableFuture` / 自建线程池即违规；
6. 不存在任何 Spring AI 自动工具执行路径。

这六条来自 Constitution 八原则和各节反复钉的设计，是"整体还是 OryxOS"的最小判据。

### H5 实现质量（硬门禁为主）

- 异常不吞：每个 catch 要么落审计/日志（人类可读的中文 `error_message`）要么上抛，必占其一；课件 harness 列出的失败用例必须实现。
- 不过度设计：文档没规定的抽象层不建，单实现不建接口（文档规定的"接口墙"除外），不留死代码。
- 注释只写"为什么"（约束与取舍），不复述代码；命名与文档词汇一致。
- 避开 P3C/ASM 解析不了的 Java 18+ 语法形态（已实踩：增强 switch 的 `default ->` 误报）——静态检查是门禁的一部分，解析失败等于构建失败。

### H6 完成定义与反作弊（硬门禁收口）

**任务级完成** = 实现与对应测试一起落地、该模块测试绿——红了当场修，不攒。

**节级完成** = 六项证据齐全，缺一不算：

1. `mvn clean verify` 全绿（含 P3C / SpotBugs / FindSecBugs / PMD）；
2. 课件"验收 harness"映射表里的每个测试类存在且非空，关键回归测试逐个对得上号；
3. "本节交付物"逐项存在性核对通过（ls / grep 证据写入报告）；
4. 前序节测试回归全绿；
5. H4 六条不变量自查通过；
6. 产出验收报告：以上证据 + 剩余人工项清单。

**反作弊条款**：不得通过删断言、`@Disabled`、放宽阈值让测试变绿——实现错就修实现；认为测试本身错，停下报告。未全绿不得宣称完成。

## 四、门禁挂在流程的哪个位置

| 流程步骤 | 生效的门禁 |
|---|---|
| 开工 | H0 必读与依赖检查 |
| /speckit-specify、/speckit-plan | H1 设计保真（喂料只来自课件与技术方案） |
| /speckit-tasks 之后 | **固定软停点**：任务 ↔ 交付物清单自动比对，差异报告，等人确认再 implement |
| /speckit-implement 每个任务 | H3 API 核实（写前）；任务级 DoD（写后）；H5 随写随查 |
| 触发设计变更的任何时刻 | H1/H2/H3 软门禁：停下报告 |
| 一节收尾 | H2 回归 + H4 自查 + H6 六项证据 + 验收报告 |
| 收尾之后 | **不自动 commit / push / package.sh**——同步时机由人决定 |

## 五、两种 harness 的分工

这个项目里"harness"一词出现在两处，分工不同、互相成就：

- **测试 harness**（各节课件"验收 harness"部分）：具体到某节的测试类清单和关键回归测试——它是**硬门禁的执行体**。"想清楚"阶段点过名的每个坑，都有一个回归测试把它永久钉死。
- **提示词门禁**（本文 H0～H6）：跨节通用的行为约束——它规定 AI **在什么情况下必须停**、什么状态才许说"完成"。测试管"做出来的对不对"，提示词门禁管"做的过程越没越界"。

两者合起来才是第 6 节说的完整 Harness：**SDD（课件+技术方案）给目标，Harness（测试+门禁）给边界，Loop（Spec-Kit 流程）给可控执行。**

## 六、这套设计的可迁移性

去掉 OryxOS 的具体名词，骨架适用于任何"AI 执行、人握方向"的项目：

1. 指定唯一权威设计源，冲突不自行仲裁；
2. 公共概念白名单制——交付物清单之外的对外概念一律停下报告；
3. 已定字面量逐字保真，禁止顺手优化；
4. "不做清单"与"要做清单"同等硬度；
5. 提炼全局不变量，每次交付前逐条自查；
6. 完成定义证据化 + 反作弊条款；
7. 能机器判的绝不留给人，机器判不了的绝不自行发挥。
