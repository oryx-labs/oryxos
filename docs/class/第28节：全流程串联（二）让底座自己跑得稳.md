# 全流程串联（二）：让底座自己跑得稳——定时任务、重启恢复、多 Agent 并存

上一节把人推主干拉通了：一句话进来，八站走完，账全对得上，三面同源。这一节接着补另外半边——让底座在**没人看着的时候也持续正确**。同样是集成课，不引入新模块；交付物依旧不是"某个功能"，而是"**底座跑得稳**"这件事本身：定时任务自己到点触发、重启后不失忆、多个 Agent 并存互不打架。

---

## 一、本节目标：让底座自己跑得稳

先把终点说清楚，一句话：

> **上一节验证的是"有人问、答得对"；这一节验证的是"没人问、也照样干活；重启了、也不失忆；多个 Agent、互不打架"。**

这三件事，正是 Agent OS 和"一个 Agent Demo"的分水岭——Demo 只要在演示的那几分钟里活着就行，底座却要在没人盯着的时候也持续正确。把目标拆成三根能验证的支柱：

- **定时任务**：一个配了定时的 Profile，到点**自己**跑完"查天气 → notify 推送"，webhook 那头真收到，三张表痕迹不多不少，定时那条会话被复用（不是每次触发都新建一条）；
- **重启恢复**：`kill` 掉进程再 `serve` 起来，会话、记忆、定时任务、审计记录**四样全部原样回来**；
- **多 Agent 并存**：两个差异明显的 Profile 跑在同一个实例上，**工具、会话、定时三条隔离边界**都守得住。

三根支柱同时立住，底座才算"跑得稳"。

![定时链路五站：定时触发、统一入口、ReAct、notify 推送、落库](../../website/public/images/class-28-1.svg)

**为什么现在做、用什么方法。** 上一节把人推主干串通了，但那只证明"被人调用时是对的"。底座真正的价值，在于**没人值守时也持续正确**——而这恰恰是最容易藏 bug 的地方：状态偷偷赖在内存里、某个失败拖垮了整个调度器、两个 Agent 不小心共用了同一份默认配置。方法还是上一节那套**对账法**，一点不变：定时链路无非是人推链路换了个"触发头"（`AgentScheduler` 代替人来发起）、多了个"推送尾"（`notify` 代替屏幕输出），中间的引擎完全复用。所以我们照样拿一次真实触发，从进到出走一遍，逐张表核对痕迹。下面第二章，就是围绕这三根支柱要做的事。

## 二、围绕目标要做哪些

### 2.1 把定时链路跑通

还是上一节的对账法。配一个测试 Profile：定时设成每两分钟触发一次，消息是"查一下北京天气，把穿搭建议推送出去"，通知渠道指向一个测试 webhook。让它自己跑一轮，然后对账——这次触发应该留下、而且只留下这些痕迹：

- **`sessions`（会话表）**：渠道和用户都标着 `scheduler` 的那条会话**被复用**（第二次触发不新建，历史追加在同一条里，这是 25 节定的规则）；
- **`llm_calls`（模型调用表）**：两条（一轮决定要调工具、一轮组织答复）；
- **`tool_invocations`（工具调用表）**：两条——`http_get`（查天气）和 `notify`（推送），都成功；
- **webhook 那头**：真的收到了消息。

**重点盯两处接缝**：一是**会话复用**——连续触发两次之后，`sessions` 表里还是一条记录、只是对话历史变长了，而不是冒出两条；二是**推送的失败路径**——故意把 webhook 地址改成白名单外的域名，确认 Sandbox 拦下、`tool_invocations` 里留了一条 `success=false`、而且**调度器本身没被拖死**（下一个触发点照常到来）。

### 2.2 重启不失忆

底座的状态全部放在进程外面（SQLite 数据库 + 文件系统），进程自己不存状态——这句架构承诺，重启一次就能验证：

1. 先跑几轮对话、攒几条记忆、让定时任务至少触发过一次，然后 `kill` 掉进程；
2. 重新 `oryxos serve`，逐项核对：
   - **会话回来了**：`GET /sessions/{id}` 能查到重启前的完整历史（本来就存在 SQLite 里）；
   - **记忆回来了**：新对话里 Agent 还记得核心记忆的内容（`MEMORY.md` 是文件，天生跨重启）；
   - **定时任务回来了**：`AgentScheduler` 是启动时扫一遍 Profile 重新建起来的，不依赖任何进程内的残留——重启后它应当把定时任务重新注册、到点照常触发；
   - **审计不断档**：重启前后的 `llm_calls` 记录是连续的，没有丢掉一段。

哪一项没回来，就说明有状态偷偷留在了进程内存里——这是走向分布式之前必须掐灭的隐患（"无状态实例"是技术方案定下的前提）。

### 2.3 多 Agent 并存

"OS"在核心阶段最小的体现，就是**多个 Profile 能在同一个实例上同时用**。配两个差异明显的 Profile（比如一个只给文件工具、一个只给 HTTP 工具），交替使用，核对三条隔离边界：

- **工具隔离**：在 Profile A 的对话里，模型拿到的工具清单里**不该**出现 B 独有的工具（`tools` 字段的过滤生效了）；
- **会话隔离**：A 和 B 各自的会话互不串——会话 id 里带着 Profile 名，天然分开；但要提防哪处代码图省事用了"默认 Profile"，把两边搅到一起；
- **定时隔离**：A 的定时任务抛了异常，B 的定时任务下一个触发点照常执行（25 节的失败隔离，在多 Agent 场景下再验一遍）。

### 2.4 从"能跑"打磨到"跑得稳"

最后一轮打磨，都是小事，但件件影响 Demo 现场的体感：

- **该超时的都要超时**：模型调用、工具执行、notify 推送都要有超时上限，一处卡死不能拖垮整根虚拟线程；一次 Agent 调用整体 60 秒封顶（26 节定的）。
- **错误信息要人看得懂**：`error_message` 里存的应是人能读懂的原因（"域名不在白名单内: evil.com"），不是一坨堆栈——审计表是给人查的。
- **一次触发在日志里是一条清晰主线**：一次对话或一次定时触发，能靠会话 id 从日志里串出完整过程，不用在几百行里捞。
- **外部 MCP 挂了不能拖垮自己**：外部 MCP server 连不上，就记条 WARN、跳过它提供的工具，OryxOS 自己照常启动——**别人的可用性，不能变成自己的可用性。**

### 2.5 补一个对账要用的查询接口：让定时任务对外可见

沿用上一节的做法——对账要能**自动验证**，前提是每样痕迹都有个 HTTP 接口查得到。把三根支柱要断言的东西，逐条对着现有接口查缺：

| 要断言的 | 靠哪个接口 | 现状 |
|---|---|---|
| 会话被创建/复用、能列出 | `GET /api/v1/sessions` + `/{id}` | 27 节补的列表 + 26 节的按 id 查 |
| 记忆写得进、查得到 | `GET /api/v1/memory` | 26 节已有 |
| 多个 Agent（Profile）都在 | `GET /api/v1/profiles` | 26 节已有 |
| **定时任务注册了、重启后回来了** | `GET /api/v1/schedules` | **缺——本节补** |

前三样都齐了。**唯一缺的是"定时任务的可见性"**：`AgentScheduler` 现在只是把 cron 丢给调度器，自己没留一份"已注册任务"的清单——外界只能靠翻启动日志来判断"定时到底回来没有"，测试没法稳定断言。本节把它补上：

- **`GET /api/v1/schedules`** —— 列出当前已注册的定时任务：Profile 名、任务 id、cron 表达式、时区、**下次触发时刻**、是否启用。
  - **要改的地方**：`AgentScheduler.registerAll()` 注册时，把每条"（Profile、定时配置、下次触发时刻）"记进一张只读清单，再加一个 `registered()` 访问器（沿用 25 节既有类的扩展方式，不动调度逻辑本身）；`oryxos-web` 加一个 `ScheduleApiController` 和一个摘要 DTO 来读它。
  - 这样"重启后定时任务回来了"，就从**翻日志**变成 `GET /api/v1/schedules` 一次断言（清单非空、下次触发时刻在未来）；同时也给管理台留了一个未来"定时任务"页的数据来源。

### 2.6 为 Demo 备好环境（前置条件清单）

31 节那两个 Demo 要用的环境，这一节末尾一次配齐、逐项打勾：

![六项前置条件：白名单、通知渠道、新闻 MCP、记忆偏好、定时配置、跨重启](../../website/public/images/class-28-2.svg)

> **最容易自己绊倒自己的一条：白名单会拦下你自己的 Demo。** 24 节的 Sandbox 默认全部拒绝，`notify` 推送（19 / 24 节接线）和 `http_get` 查天气都得先过 `http.allowed_domains`。所以配环境时，必须把三样域名显式加进白名单，否则 31 节的 Demo 全是 `tool_invocations` 里的 `success=false`：① 天气源 `api.open-meteo.com`（Demo 一）；② 团队通知 webhook 的域名（如飞书 `*.feishu.cn`、企业微信 `qyapi.weixin.qq.com`，按你实际用的渠道填）；③ 如果日报靠 `http_get` 拉新闻，再加上新闻源的域名。`file` / `shell` 两类白名单这两个 Demo 用不上，保持"全部拒绝"即可。另外确认 26 节已经排除了 `OpenAiAutoConfiguration`——否则 `serve` 根本起不来，这轮串联无从谈起（26 节决策四）。

## 三、怎么验收：把对账固化成测试

手工对账查完就忘——尤其重启那组，手动做一次就得五分钟。这一节把它固化成两个集成测试（都打 `@Tag("integration")`，需要真 key、CI 默认跳过、手动或发版前跑）：`SchedulerFlowIT`（定时链路）和 `RestartRecoveryIT`（重启恢复）。harness 好不好只有一条标准：**它替你把"定时自己跑通 + 重启不失忆 + 多 Agent 不打架"自动断言一遍。** 三根支柱对应三组断言：

### 支柱一 · 定时任务自己跑通（SchedulerFlowIT）

1. 配测试 Profile：定时、消息"查北京天气推送穿搭"、通知渠道指向测试 webhook（测试里起一个接收端把推送收下来）；
2. 触发一次（可以直接调 `AgentScheduler.runOnce` 免得等两分钟，或者真等 cron）→ 断言：
   - `sessions`：`scheduler` 那条会话**被复用**（连续触发两次仍是一条，25 节规则）；
   - `llm_calls` 恰 2 条、`tool_invocations` 恰 2 条（`http_get` + `notify`）、全部成功——不多不少；
   - 测试 webhook **真收到**了消息体；
3. 失败路径：把 webhook 域名改到白名单外 → 断言 Sandbox 拦下、`tool_invocations` 留了 `success=false`、**调度器没死**（下个触发点照常）。

### 支柱二 · 重启不失忆（RestartRecoveryIT）

1. 先跑几轮对话 + 攒记忆 + 触发定时一次，然后 `kill`；
2. 重启 `serve` → 断言（尽量走 HTTP，好断言）：
   - `GET /sessions/{id}` 查到重启前的完整历史；
   - `GET /api/v1/memory` 查到核心记忆；
   - **`GET /api/v1/schedules`（本节新增）列出重新注册的定时任务、下次触发时刻在未来**；
   - 审计连续：`llm_calls` 跨重启不断档（查库确认）。

### 支柱三 · 多 Agent 不打架

1. 配两个差异 Profile（A 只给文件工具、B 只给 HTTP 工具）；`GET /profiles` 里两个都在；
2. 交替使用 → 断言三条边界：
   - **工具隔离**：A 的对话里模型拿到的工具清单里没有 B 独有的工具；
   - **会话隔离**：`GET /sessions` 里 A、B 各自的会话（id 带 Profile 名）互不串；
   - **定时隔离**：A 的定时任务抛异常，B 的下一个触发点照常执行。

### 验收清单（每条都成立才算达标）

- **定时**：测试 Profile 到点自己跑完"查天气 → 推送"，webhook 收到，逐表对账不多不少，会话复用；
- **定时失败隔离**：notify 打到白名单外时被拦、记 `success=false`，调度器不死；
- **重启**：`kill` 再起，`GET /sessions/{id}`、`GET /api/v1/memory`、`GET /api/v1/schedules`、审计四样全部原样恢复；
- **多 Agent**：两个 Profile 并存，工具、会话、定时三条隔离边界都验证过；
- **稳定性**：超时兜底、错误信息可读、外部 MCP 挂了不拖垮启动，各验一次；
- **Demo 前置**：三样域名进了 `http` 白名单、通知渠道配好、`OpenAiAutoConfiguration` 已排除；
- **不回退**：`mvn clean verify` 全绿，`SchedulerFlowIT` / `RestartRecoveryIT` 打 `@Tag("integration")` 手动能跑通。

### 本节交付物

- **代码**：`AgentScheduler` 加"已注册任务清单" + `registered()` 只读访问器（core 既有类扩展，不动调度逻辑）；`oryxos-web` 新增 `ScheduleApiController`（`GET /api/v1/schedules`）+ 摘要 DTO。
- **测试（harness）**：`SchedulerFlowIT`、`RestartRecoveryIT`（都打 `@Tag("integration")`），覆盖上面三根支柱 + 失败隔离。
- **配置**：Demo 前置环境——三样域名进 `http.allowed_domains`、通知渠道配好、测试 Profile 的定时配置。
- **文档**：README 端点表补上 `GET /api/v1/schedules`。

到这里，底座部分（第一部分）就全部完工、而且跑得稳了：人推答得对、定时自己跑、重启不失忆、多 Agent 不打架。但请注意一个事实：我们到现在还没真正"定义"过一个业务 Agent——测试用的 Profile 都是顺手配的。下一节讲这门课最关键的一次视角转换：**底座是底座，Agent 是 Agent**——用 Skill 把一个业务 Agent 声明式地定义出来。
