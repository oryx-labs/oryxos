# 任务：从零构建 OryxOS 项目（企业级 Agent OS）

> 本文件由第一轮 45 条交互式提示词归纳而成，重组为可供自主 loop 编程 agent 一次性理解并执行的结构化任务。

## 角色

你是一个自主编程 agent。请按下方规格，从空仓库开始，分阶段构建整个 OryxOS 项目，每完成一个阶段就自检（编译 / 运行 / 构建通过）再进入下一阶段，全程不需向我确认，遇到歧义时按工程惯例选最合理方案并记录决策。

## 项目背景

OryxOS 是基于 Java 21 + Spring Boot 3.x 的企业级 Agent OS，作为统一底座运行多个业务 Agent，共享：渠道接入、模型路由、工具调用、记忆系统、沙箱执行。

权威需求来源：`docs/` 下的设计文档（DemandAnalysis / TechnicalSolution / IndustryResearch / AiProgrammingGuide / oryxos.md / what.md）。第一步必须先完整读懂这些文档，它们定义了项目要做的全部事情。

## 阶段一：文档与规范

1. 读懂 `docs/` 下所有设计文档，提炼项目目标、架构、约束。
2. 把这些文档整理为格式规范、可读性好的 Markdown（统一标题层级、表格、代码块）。
3. 基于文档生成 `CLAUDE.md`（项目指南：技术栈、模块结构、不可违背原则、数据模型、ReAct Loop 机制、Tool 体系、API、CLI、四周节奏、常见陷阱）。
4. 基于文档生成 `README.md`，参照标准开源项目结构（徽章、简介、架构图、快速开始、模块说明、License）。

## 阶段二：视觉资产

5. 画 Architecture 架构图，SVG 或 PNG 输出到 `docs/images/`，元素不得重叠、不拥挤、层次清晰；README 的架构图引用这张图。
6. 设计 OryxOS Logo（SVG，`docs/images/logo.svg`），以橙色为主色调，纯字形（不带副标题文字），尺寸适配 README 居中展示与网站左上角小图标两种场景。

## 阶段三：工程骨架

7. 用 Maven 多模块初始化项目，严格按文档划分 9 个模块（oryxos-core / provider / memory / tool / channel-cli / web / storage / cli / boot），模块间通过接口解耦，必须能通过 `mvn clean package` 编译打包。
8. `oryxos-boot` 提供 Spring Boot 主类，可 `java -jar` 运行。
9. `oryxos-cli` 提供 Picocli 主入口（main 函数），可运行并打印 OryxOS 版本信息。

## 阶段四：官网（website/，VitePress）

10. 用 VitePress 搭建官网，自主设计有特点的首页风格（不要套默认模板）。
11. 主色调橙色、整站纯黑背景（顶部导航与页脚也是黑底，不留白条）。
12. 中英文双语（zh / en）。
13. `base` 配置为 `/oryxos/`（GitHub Pages 项目页），部署域名 `oryxos.robustmq.com`，GitHub 仓库 `https://github.com/oryx-labs/oryxos`。
14. 内容与 README / 设计文档保持一致；改版时同步更新三处（README、官网首页、doc）。

## 阶段五：工程化与发布

15. 提供 `scripts/package.sh`（参考标准打包脚本：编译、打包、上传远程），上传时排除所有 `target/` 产物。
16. `.gitignore` 忽略 `website/.vitepress`（cache/dist）与各模块 `target/`。
17. 配置 GitHub Pages：新增 `.github/workflows/deploy-pages.yml`（VitePress 构建 → `actions/upload-pages-artifact` → `actions/deploy-pages`，权限 `pages:write` + `id-token:write`，`base=/oryxos/`），并在 Settings→Pages 将 Source 设为 GitHub Actions。确保 CSS/JS 资源路径与 base 一致，避免线上 404。

## 全局约束（不可违背）

- Java 21，不得降版本；同步执行模型，不引入 Reactor / WebFlux / CompletableFuture。
- ReAct Loop 自实现，禁用 Spring AI 自动 tool 执行（Spring AI 只用于协议转换 + `@Tool` schema 生成）。
- 多 Provider 用显式 `Map<String, ChatModel>` 映射，不靠类型扫描。
- 沙箱用 `SandboxChecker` 白名单（路径 / 命令 / 域名），不使用 `SecurityManager`。
- 审计表 `tool_invocations` / `llm_calls` 核心阶段即写入 SQLite。
- 敏感配置（API key、token）只走环境变量，绝不明文写进 YAML / 代码 / 提交记录。

## 验收

- `mvn clean package` 通过；`java -jar` 启动 boot 可运行；cli 可打印版本。
- 官网本地 `npm run docs:dev` 与 `npm run docs:build` 均成功，线上 Pages 无资源 404。
- README、官网、docs 三处内容一致，架构图与 logo 正确引用。
