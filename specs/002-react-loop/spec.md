## Clarifications

### Session 2026-07-10

- Q: 会话历史截断中"一轮"如何界定？ → A: 以用户消息为界——一轮 = 一条用户消息及其后到下一条用户消息前的全部模型响应与工具结果；截断保留最近 N 个完整轮次（默认 20）。课件与技术方案均用"最近 N 轮"表述，此为其可测试化定义（默认自答，停点供确认）。
- Q: 处理中途抛异常时，已累积的会话历史是否持久化？ → A: 不持久化——持久化只发生在循环正常返回之后，异常路径只保证清理 Agent 上下文后原样抛出（与课件 AgentService 骨架一致：save 在 try 内、run 之后）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 多轮"想—做—看"完成一件事 (Priority: P1)

用户问"看看今天天气，帮我决定穿什么"。Agent 不能一句话糊弄过去：它先要求调用天气查询工具，系统执行工具、把结果交还给它，它再根据结果给出穿搭建议。整个过程在一个循环里自动推进，用户只看到最终答复。

**Why this priority**: 这是 ReAct 存在的全部意义——没有这个循环，系统就只是个一问一答的 chatbot，Demo 一（每日天气）无法成立。

**Independent Test**: 用测试替身模拟模型行为（第一轮要求调工具、第二轮给出最终答复），验证循环恰好执行了两轮模型调用、一次工具执行，且最终返回第二轮的答复文本。

**Acceptance Scenarios**:

1. **Given** 模型第一轮要求调用天气工具、第二轮直接给出答复，**When** 用户发起"查天气穿衣"请求，**Then** 系统执行该工具一次，并把第二轮的答复文本返回给用户。
2. **Given** 模型第一轮就直接给出答复（未要求调用任何工具），**When** 用户发起请求，**Then** 循环一轮即收尾，返回该答复，不发生任何工具执行。
3. **Given** 模型某轮同时要求调用多个工具，**When** 该轮结束，**Then** 各工具依次逐个执行（不并行），结果全部交还给模型进入下一轮。

---

### User Story 2 - 死循环兜底：转满最大轮数强制停 (Priority: P1)

模型异常地每轮都要求调工具、永不收敛。系统不会陷进死循环：转满每个 Agent 配置的最大轮数（默认 10）后强制停下，并在答复中明确告知已达上限。

**Why this priority**: 不设兜底，一个异常会话就能耗尽 token 和线程——这是设计时就必须定死的边界，不是事后补丁。

**Independent Test**: 测试替身让模型每轮都要求调工具，验证模型恰好被调用了最大轮数次（一轮不多），且返回文本包含"达到最大轮数"。

**Acceptance Scenarios**:

1. **Given** 模型每轮都要求调工具、永不收敛，且该 Agent 最大轮数配置为 10，**When** 用户发起请求，**Then** 模型恰好被调用 10 次后循环强制收尾，答复包含"达到最大轮数"。
2. **Given** 某 Agent 配置最大轮数为 5，**When** 同样场景，**Then** 恰好 5 轮后停（上限按 Agent 配置生效，不是写死的全局值）。

---

### User Story 3 - 全程可审计：每轮都留痕 (Priority: P2)

审计员事后查某次会话：每一轮模型说了什么、要求调了哪个工具、参数和结果是什么、成功还是失败、耗时多久，全部有记录可查。

**Why this priority**: 可审计是本系统的核心差异化能力；模型调用审计上一特性已交付，本特性补齐工具调用这半边并保证会话累积不缺轮。

**Independent Test**: 执行一次含工具调用的处理后，检查会话历史包含每轮模型响应与工具结果；工具审计记录不论成败都存在，失败记录带 success=false 和原因。

**Acceptance Scenarios**:

1. **Given** 一次含工具调用的多轮处理，**When** 处理结束，**Then** 会话历史里能按顺序看到：用户消息、每轮模型响应、每次工具结果。
2. **Given** 某次工具执行成功，**When** 查审计记录，**Then** 有一条 success=true 的记录，含工具名、入参、结果、耗时，按会话关联。
3. **Given** 某次工具执行抛出异常，**When** 查审计记录，**Then** 有一条 success=false 的记录，带失败原因；异常不被吞掉，结果以失败形式交还模型进入下一轮。

---

### User Story 4 - 统一处理入口与 Agent 上下文生命周期 (Priority: P2)

命令行、Web、定时任务三种触发源最终都走同一个处理入口。处理期间，"当前是哪个 Agent"的配置对工具执行可见（后续通知类工具要靠它找到自己的推送渠道）；处理结束时该上下文必须被清理——即使处理中途抛异常也不例外；处理完成后会话历史被持久化。

**Why this priority**: 上下文清理不做对，单请求测试永远不报错，只在并发复用线程时串号——最阴险的一类 bug，必须在本特性显式钉死。

**Independent Test**: 处理期间断言可取到当前 Agent 配置；让循环抛异常，断言异常抛出后上下文已被清空；正常结束后断言会话保存动作发生。

**Acceptance Scenarios**:

1. **Given** 一次正常处理，**When** 工具在处理期间读取当前 Agent 配置，**Then** 取到的是本次会话对应的 Agent 配置。
2. **Given** 循环处理中途抛出异常，**When** 异常传播出处理入口后，**Then** 当前 Agent 上下文已被清空（下一个复用此线程的请求不会拿到别人的配置）。
3. **Given** 一次正常处理结束，**When** 处理入口返回，**Then** 累积完的会话历史已被持久化。

---

### User Story 5 - 上下文组装：新鲜、有序、可控长度 (Priority: P2)

每轮发给模型的上下文按固定四部分顺序组装：角色与启动上下文（末尾附当前日期时间）、长期记忆（未启用则跳过）、会话历史（只留最近 N 轮）、可用工具列表。角色/启动/技能文件每次都重新读取——用户改完文件，下一轮立即生效；引用的文件缺失不许静默跳过。

**Why this priority**: 上下文撑爆（坑二）和"人格悄悄丢了"（静默跳过缺失文件）都是最难查的软故障，必须在组装层拦死。

**Independent Test**: 构造超过 N 轮的会话历史，验证组装结果只含最近 N 轮；修改上下文文件后再次组装，验证读到新内容；引用不存在的技能文件时报错、启动文件缺失时留下告警。

**Acceptance Scenarios**:

1. **Given** 会话历史超过该 Agent 配置的保留轮数（默认 20），**When** 组装上下文，**Then** 只包含最近 N 轮，更早的被截断。
2. **Given** 一次组装完成后用户修改了角色/技能文件，**When** 下一轮再组装，**Then** 立即读到新内容（无缓存）。
3. **Given** Agent 显式引用的技能文件不存在，**When** 组装，**Then** 报错并点名缺失文件；**Given** 启动上下文文件缺失，**Then** 至少留下告警，不静默跳过。
4. **Given** 任意一次组装，**When** 检查系统上下文部分，**Then** 末尾含当前日期时间（模型自己不知道今天几号）。

---

### Edge Cases

- 模型一轮同时要求调多个工具 → 逐个顺序执行，结果全部回填后再进下一轮（不并行）。
- 工具执行抛异常 → 审计记录 success=false 带原因；失败结果照样交还模型，由模型决定下一步；异常不中断循环、不被吞掉。
- 模型响应既无文本也无工具请求 → 视为可收尾，返回空文本按"最终答复"处理。
- 最大轮数配置为 1 → 第一轮无论是否要求调工具，调完模型（并执行完该轮工具）后即达上限。
- 会话历史恰好等于 N 轮 → 不截断；N+1 轮起截掉最早的。
- 长期记忆能力尚未交付（属后续特性）→ 该部分暂为空、跳过，组装顺序保持四段framework不变。
- 工具审计写入自身失败 → 记录错误日志但不阻断工具结果返回（与模型调用审计同口径）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 循环单元 MUST 只做调度——转圈、判断该不该停、把每轮结果攒起来；上下文组装、模型调用、工具执行 MUST 各由专门单元承担，循环里不得内联这些逻辑。
- **FR-002**: 停止条件 MUST 双保险：模型某轮未要求调用工具即收尾返回其答复；否则最多转该 Agent 配置的最大轮数（默认 10），转满强制收尾且答复中明示已达上限（含"达到最大轮数"字样）。
- **FR-003**: 每轮模型响应与每次工具执行结果 MUST 累积回会话，顺序保持，事后可完整回放；下一轮组装 MUST 能看到之前所有轮次。
- **FR-004**: 每轮上下文 MUST 按固定四部分顺序组装：①角色与启动/技能上下文（末尾附当前日期时间）②长期记忆（未启用跳过）③会话历史（只留最近 N 轮，Agent 可配默认 20；一轮 = 一条用户消息及其后的全部响应/工具结果，见 Clarifications）④当前可用工具列表。
- **FR-005**: 工具执行 MUST 只有唯一一条路径；执行前 MUST 过沙箱/白名单检查位（检查规则由后续特性提供，本特性留位）；每次执行不论成败 MUST 落工具审计（tool_invocations）：工具名、入参、结果、success 标识、失败原因、耗时，按会话关联；工具异常 MUST 转为失败结果交还模型，不得吞掉也不得中断循环。
- **FR-006**: 系统 MUST 提供唯一处理入口供所有触发源使用；处理期间当前 Agent 配置 MUST 对工具执行可见；处理结束（含异常路径）MUST 清理该上下文；仅正常结束后 MUST 持久化会话历史（异常路径不持久化，见 Clarifications）。
- **FR-007**: 角色/启动/技能上下文文件 MUST 每次组装重新读取不缓存；Agent 显式引用的技能文件缺失 MUST 报错点名；启动上下文文件缺失 MUST 至少告警，不得静默跳过。
- **FR-008**: 模型调用 MUST 携带会话标识，复用上一特性交付的模型调用审计（llm_calls）；本特性不重复实现模型审计。

### Key Entities

- **会话（Session）**: 一次对话的全部状态——会话标识、所属 Agent 名、按序累积的消息记录（用户消息 / 模型响应 / 工具结果）。是循环的输入与累积容器，处理完成后整体持久化。
- **工具调用审计记录（ToolInvocation）**: 一次工具执行的完整留痕——所属会话、工具名、入参、结果、success 标识、失败原因、耗时、发生时间。成败都写。
- **Agent 配置（Profile，上一特性已交付）**: 提供最大轮数、历史保留轮数、角色设定、启动/技能文件引用、可用工具清单。
- **上下文（组装产物）**: 每轮发给模型的完整输入，四部分有序拼接，非持久化、轮轮重建。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: "查天气→给建议"类两轮任务在一次请求内自动完成，用户只看到最终答复，无需二次输入。
- **SC-002**: 模型永不收敛时，系统恰好在配置的最大轮数处停下（一轮不多），答复明示已达上限——该场景 100% 可复现、有自动化回归钉死。
- **SC-003**: 任一次处理结束后，审计视角能完整回放该会话：每轮模型说了什么、每次工具调用的入参/结果/成败/耗时，零缺轮。
- **SC-004**: 处理中途抛异常的场景下，Agent 上下文泄漏率为零（异常路径回归测试钉死）；并发复用线程时不发生配置串号。
- **SC-005**: 用户修改角色/技能文件后，下一轮对话即生效，无需重启（自动化回归验证无缓存）。
- **SC-006**: 会话历史超限时上下文长度受控：组装产物只含最近 N 轮，N 由每个 Agent 自行配置。

## Assumptions

- 上一特性（Provider）交付物为直接上游：按 Agent 配置路由调用模型（携带会话标识供审计关联）、返回文本与工具调用请求；`max_iterations` / `max_history_turns` 配置项已就位（默认 10 / 20）。
- 工具调用审计口径与上一特性 `llm_calls` 完全一致：手工建表脚本、成败都写、审计写入自身失败只记日志不阻断。
- 工具的真实实现（文件/Shell/HTTP）属第 20 节，本特性以测试替身验证调度与审计路径；沙箱检查规则属第 23/24 节，本特性只留检查位。
- 长期记忆属第 22 节，本特性组装时该部分为空跳过，但四段顺序框架先立好。
- 会话的持久化存储与跨渠道会话管理属第 18 节；本特性只要求"处理结束后发生持久化动作"，以最小抽象对接。
- 明确不做（边界）：工具并行调用、Agent 之间互相委托、流式输出、上下文压缩——核心阶段都不做；上下文长度先用"只留最近 N 轮"顶着。
