## User Scenarios & Testing *(mandatory)*

### User Story 1 - 对话里记住、跨会话在场 (Priority: P1)

用户在对话里说了一句值得长期记住的话（偏好、项目背景、关键约束）。Agent 自己判断"这该记"，主动调记忆工具写入，并可指定它属于"核心"还是"归档"。开一个全新会话时，系统提示里依然带着核心记忆——"始终在场"，Agent 一上来就记得你是谁。

**Why this priority**: 这是 Memory 存在的全部意义；没有它，Agent 每次新会话都是失忆的陌生人，Demo 二（日报体现用户偏好）无从谈起。

**Independent Test**: 写入一条核心记忆、若干归档记忆，构造一个新会话取门面的"待注入内容"，断言核心记忆完整在内、会话历史一并组合。

**Acceptance Scenarios**:

1. **Given** Agent 判断某句话值得记，**When** 调记忆工具写入（指定核心），**Then** 该内容进入核心区。
2. **Given** 已有核心记忆，**When** 组装任意新会话的 Prompt，**Then** 核心记忆完整出现在待注入内容里（不检索、不截断）。
3. **Given** 写入一条记忆后同一进程内，**When** 立刻读取/检索，**Then** 立即可见（无缓存）。

---

### User Story 2 - 换后端零改上层：接口墙的兑现 (Priority: P1)

运营方随记忆规模演进想换存储：量小用文件、量大换结构化本地库、真需要智能记忆换外部记忆服务。切换只改一个配置项，门面以上（Prompt 组装、记忆工具、循环）一行代码都不用动，对话体感完全不变。

**Why this priority**: 这是本节最想证明的东西——"接口不变、实现随便换"不是口号。三档递进让这道墙的价值看得见摸得着。

**Independent Test**: 同一套契约断言参数化跑三档后端全过；改配置装配不同后端，上层调用代码与断言零改动。

**Acceptance Scenarios**:

1. **Given** 三档后端各一个实例，**When** 对每一档跑同一组契约断言，**Then** 全部通过——写后立读、截断保核心、scope 路由、检索只搜归档。
2. **Given** 配置指定某一档，**When** 装配记忆层，**Then** 门面注入的是该档实现；换配置换实现，门面签名与上层调用不变。
3. **Given** 任一后端，**When** 上层通过门面读写记忆，**Then** 上层无法感知底层是文件、本地库还是外部服务。

---

### User Story 3 - 核心与归档分治，截断不伤核心 (Priority: P1)

长期记忆分两类：核心记忆（用户是谁、项目背景、关键偏好，永远完整在场）和归档记忆（量大的历史事实，超限截断）。无论哪一档后端，截断只作用在归档区，核心记忆一字不能少。

**Why this priority**: "核心记忆始终在场"是体感底线；截断逻辑一旦连累核心区，就违背了它的定位——这是最容易被"优化"坏的一条，必须钉死。

**Independent Test**: 灌入一条核心记忆 + 远超归档上限的归档记忆，读取后断言核心记忆完整、归档区最早的被裁、最近的保留。

**Acceptance Scenarios**:

1. **Given** 一条核心记忆 + 归档记忆灌到远超上限，**When** 读取长期记忆，**Then** 核心记忆完整、归档区只保留最近部分。
2. **Given** 写入时指定 scope，**When** 读取分区，**Then** 内容落在指定区块（核心/归档路由正确）。
3. **Given** 关键词检索，**When** 执行，**Then** 只在归档区匹配（核心区本就全量注入、不参与检索）。

---

### User Story 4 - 记忆工具对后端无感 (Priority: P2)

Agent 通过两个内置工具读写长期记忆：记一条（带可选的核心/归档参数，缺省归档）、按关键词查一下（未命中返回友好提示而非报错）。这两个工具对"底下是哪档后端"完全无感。

**Why this priority**: 工具是 Agent 触达记忆的唯一入口；"缺省归档""未命中不报错"是保证 Agent 用起来顺、不被异常打断的细节。

**Independent Test**: 记一条不带 scope 断言进归档；查一个不存在的词断言返回友好提示、不抛异常。

**Acceptance Scenarios**:

1. **Given** 调记一条工具但不指定 scope，**When** 写入，**Then** 默认落归档区。
2. **Given** 按关键词查但无命中，**When** 执行，**Then** 返回友好提示文本，不抛异常。
3. **Given** 底层从一档换到另一档，**When** 用同样方式调这两个工具，**Then** 行为一致、无需改工具代码。

---

### Edge Cases

- 记忆文件/表/服务为空时读取 → 返回空内容（含空的核心区），不报错。
- scope 参数传入非法值 → 记一条工具明确报错点名，不静默落错区。
- 归档区恰好等于上限 → 不截断；超过一点才裁最早的。
- 外部集成档的服务不可达 → 该档读写以明确错误结束（不静默吞），不影响其它档的选型；默认档是本地档，不受外部服务影响。
- 关键词命中核心区内容但不在归档区 → 检索不返回它（核心区不参与检索，本就全量在场）。
- 同一进程内写入后未重启即读取 → 立即可见（契约一：不缓存）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供统一记忆门面，对上层只暴露三件事：取"待注入 Prompt 的记忆内容"、记一条、按关键词查；上层 MUST 只依赖门面，不直接触底层存储。
- **FR-002**: 长期记忆 MUST 抽成可插拔后端接口（追加、加载、按关键词检索三方法），读写契约与具体存储解耦。
- **FR-003**: 核心阶段 MUST 一次交付三档后端：文件式（分区文件，默认，零依赖）、结构化式（按条入本地库，零外部依赖）、外部集成式（自托管外部记忆服务，远程调用）；MUST 靠一个配置项选装配哪档，换档时门面以上零改动。
- **FR-004**: 所有后端 MUST 遵守四条契约：①不缓存（写后立读立即可见）；②核心记忆永不被截断（截断只作用归档区）；③写核心/归档由 Agent 经参数显式指定；④检索为简单关键词匹配。长期记忆 MUST 分核心/归档两类，核心记忆完整拼进每次上下文。
- **FR-005**: MUST 提供两个内置记忆工具（记一条、按关键词查）暴露给 Agent；记一条带可选核心/归档参数、缺省归档；未命中查询返回友好提示不抛异常；工具对底层后端无感。
- **FR-006**: Prompt 组装 MUST 经门面把长期记忆（核心区+归档区）与会话历史一起注入 system prompt；会话记忆 MUST 复用既有会话存储。
- **FR-007**: 外部集成档的服务地址与凭证 MUST 走环境变量占位、不落明文；且 MUST 为可自托管方案以满足数据不出域。

### Key Entities

- **记忆门面（MemoryService）**: 上层唯一入口；组合长期记忆与会话历史；三方法（buildContext / remember / recall）。
- **长期记忆后端（LongTermMemoryStore，可插拔）**: append / load / recallByKeyword；三档实现共守四契约。
- **记忆分区（MemoryScope）**: 核心 / 归档两类；决定写入区块与截断/检索行为。
- **记忆条目**: 一条长期记忆（内容 + 所属分区 + 时间）；文件档是分区里的一行、结构化档是一条记录、外部档是外部服务里的一条 memory。
- **记忆工具（MemoryTools）**: save（记一条，带 scope）、recall（按关键词查）两个内置工具。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一套契约断言参数化跑**三档后端 100% 全过**（写后立读、截断保核心、scope 路由、检索只搜归档）。
- **SC-002**: 换后端只改一个配置项，门面以上代码与测试断言改动量为**零**。
- **SC-003**: 核心记忆在归档区灌到远超上限时**完整留存率 100%**、归档区只保留最近部分。
- **SC-004**: 记忆工具在 scope 缺省、关键词未命中两种情况下均**不抛异常**、行为可预期。
- **SC-005**: 记忆数据默认**不出本地**（文件/本地库）；外部档为自托管、地址凭证零明文。

## Assumptions

- 第 18 节会话存储（SessionManager/Session）为会话记忆来源；第 17 节 PromptBuilder 是长期记忆注入的集成点、ProfileContext 提供当前 Agent（外部档按当前 Agent/用户组织记忆作用域）。
- 第 20 节 ToolRegistry 与 @Tool 注解管道用于注册两个记忆工具，与其它内置工具一视同仁；`save_memory`/`recall_memory` 是 20 节内置九工具面里预留、本节落地的工具名。
- 结构化档复用既有 SQLite（手工建表脚本，与既有审计/会话表同口径）；外部集成档使用项目锁定 BOM 内的 HTTP 客户端；默认后端为文件式。
- 明确不做（边界）：自动抽取（核心阶段由 Agent 主动调工具决定）、进程内自建向量库（语义检索交外部集成档）、情景记忆、Memory Wiki 结构化矛盾检测、记忆压缩、知识图谱后端——全部扩展阶段。
