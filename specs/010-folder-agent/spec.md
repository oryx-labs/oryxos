# Feature Specification: 插件化 Agent —— 一个目录定义一个会自己跑的 Agent

**Feature Branch**: `class-29`

**Created**: 2026-07-18

**Status**: Draft

**Input**: 第29节需求：插件化 Agent——一个目录定义一个会自己跑的 Agent，让底座能装、能跑任意多个自足的 Agent。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 往目录区丢一个目录 = 定义并装上一个 Agent (Priority: P1)

运营/开发者在固定的 Agent 目录区放一个目录：里面一份主文件（配置 + 任务指令），可选地带脚本、子指令、参考。系统一扫，这个目录就变成注册表里一个可用的 Agent——全程不写一行代码、不改一行底座。放 N 个目录就得到 N 个 Agent，互不干扰。

**Why this priority**: 这是"Agent OS"成立的最小机制——没有它，底座只是个能跑单一 Agent 的框架。是本节的 MVP，其余故事都建立在"目录已能派生成一个被登记的 Agent"之上。

**Independent Test**: 在 Agent 目录区放 1 个、再放 N 个合法目录，扫描后查询已注册 Agent 列表，应恰好出现对应数量、名字正确；不写任何代码。

**Acceptance Scenarios**:

1. **Given** Agent 目录区里有一个含合法主文件的目录，**When** 系统扫描，**Then** 注册表里出现这个 Agent（名字 = 目录名），可被列出。
2. **Given** 目录区里有 N 个合法 Agent 目录，**When** 系统扫描，**Then** 注册表里恰好出现 N 个 Agent、且不产生别的东西。
3. **Given** 一个主文件缺必填项（名字/运行配置），**When** 系统扫描，**Then** 该 Agent 加载报错并点名是哪一项缺失，但不阻断其余 Agent 与系统启动。
4. **Given** 一个主文件引用了系统未注册的能力，**When** 系统扫描，**Then** 记录一条告警，Agent 仍登记。

---

### User Story 2 - Agent 到点自己跑，正文即它的指令 (Priority: P2)

一个 Agent 目录在配置里声明"什么时候自己跑"。系统扫到就到点自动触发，无人参与地跑完"想 → 调系统能力 → 答 → 推送"，全程审计留账。触发时，这个 Agent 的正文（任务指令）作为它的说明进入模型上下文。

**Why this priority**: "会自己跑"是"Agent"区别于"一段可被调用的逻辑"的关键；定时属于 Agent 自身、不属于别处，是这套模型的核心主张之一。

**Independent Test**: 给一个 Agent 目录配上"几分钟后触发一次"的定时，等它自动触发一轮，核对：无人触发、正文进了上下文、涉外调用都有审计。

**Acceptance Scenarios**:

1. **Given** 一个带定时声明的 Agent 已登记，**When** 到达声明的时间点，**Then** 系统自动触发一轮完整运行，触发的定时参数（时刻、时区）来自这个 Agent 自己的配置。
2. **Given** 一个 Agent 被触发，**When** 组装模型输入，**Then** 这个 Agent 的正文进入 system prompt（与 Bootstrap 同层）。
3. **Given** 改了某个 Agent 目录里的正文且不重启，**When** 下一次触发，**Then** 用的是改后的新正文（无缓存、即时生效）。

---

### User Story 3 - 渐进式披露：目录里的东西用到才进上下文 (Priority: P2)

一个 Agent 的正文只放"编排"；较长的规范、参考资料、抓数脚本分成同目录的子文件。Agent 按正文指引，用系统已有的读文件/执行能力，把子指令/参考"用到才读"、把脚本"用到才跑"——脚本产出的数据进上下文、脚本代码本身不进。

**Why this priority**: 让一个 Agent 目录能承载复杂任务而不撑爆上下文，是"完整形态"的价值所在；也划清了"确定性操作交给脚本、判断交给模型"的边界。

**Independent Test**: 造一个带子指令与脚本的 Agent，触发它，核对：正文在上下文里、子指令/参考在被读之前不在上下文、脚本被执行且只有其产出进上下文。

**Acceptance Scenarios**:

1. **Given** 一个 Agent 的正文指向同目录的一份子指令/参考，**When** Agent 运行到需要它，**Then** 用系统已有的读文件能力把它读进上下文；在此之前它不预先加载。
2. **Given** 一个 Agent 目录里带一个脚本，**When** Agent 运行到需要确定性数据，**Then** 用系统已有的执行能力跑脚本，脚本产出进上下文、脚本代码不进。
3. **Given** 一个 Agent 目录里的脚本，**When** 执行它，**Then** 只允许经白名单解释器、且只能跑这个 Agent 自己目录下的脚本。

---

### User Story 4 - 运行时把一个 Agent 登记 / 注销 / 查询 (Priority: P3)

系统不只在启动时扫一次；支持在运行期间登记、注销、查询一个 Agent。运行时登记走的校验与启动扫描完全一致（同一异常、同一消息）。登记一个带定时的 Agent 时，保留一个可注销的句柄，为下一节"通过接口动态增删 Agent"铺路。

**Why this priority**: 本节先把运行时注册的能力立好（下一节才对外暴露成接口），去掉"新增 Agent 必须重启"这条尾巴；本节不做对外接口，故优先级低于前三个。

**Independent Test**: 运行期登记一个 Agent 后立即查询应可见；用一份非法配置分别走启动路径与运行时路径，两边报出的错误类型与消息应完全一致；登记带定时的 Agent 后应留有可注销句柄。

**Acceptance Scenarios**:

1. **Given** 系统已启动，**When** 运行时登记一个新 Agent，**Then** 立即可查询到它。
2. **Given** 一份非法 Agent 配置，**When** 分别经启动路径与运行时路径加载，**Then** 两条路径报出的错误类型与消息完全一致。
3. **Given** 运行时登记一个带定时的 Agent，**When** 登记完成，**Then** 系统为它的定时任务保留了一个可注销的句柄。

---

### Edge Cases

- 主文件缺必填项（名字/运行配置）→ 点名报错、跳过该 Agent、不阻断启动（US1-3）。
- 配置引用系统未注册的能力 → 告警但仍登记（US1-4）。
- Agent 目录里没有任何脚本/子指令/参考 → 只有正文也是一个合法 Agent（最简形态）。
- 脚本自己发起网络请求 → 绕过内置 HTTP 工具的域名白名单；这是明确的信任边界（"装带脚本的 Agent = 信任其作者"），本版只以"解释器 + 目录"两道白名单约束，不做容器/网络隔离。
- 两个 Agent 目录同名 → 本版不做冲突消解策略（明确不做）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 一个 Agent MUST 是 Agent 目录区里的一个目录，含一份主文件（配置部分 = 这个 Agent 自己的运行身份/模型/要用的系统能力/往哪推/什么时候跑；正文部分 = 任务指令），外加可选的子指令、脚本、参考资料；系统 MUST NOT 再要求另写一份独立的 Agent 配置文件。
- **FR-002**: 系统 MUST 扫描 Agent 目录区，把每个目录派生成一个底座认识的 Agent 配置并登记进注册表；扫描 N 个目录 MUST 恰好得到 N 个 Agent，且不产生别的东西。
- **FR-003**: 一个 Agent 的正文 MUST 在被触发时进入 system prompt（与 Bootstrap 同层）；每次组装输入都重新读取、MUST NOT 缓存（用户改完即时生效）。
- **FR-004**: 一个 Agent 的子指令/参考/脚本 MUST NOT 预先加载；Agent MUST 能按正文指引，用系统已有的读文件/执行能力把它们"用到才读/才跑"，脚本产出进上下文、脚本代码不进。
- **FR-005**: 定时 MUST 写在 Agent 目录的配置里、并被带进派生出的配置；系统 MUST 对带定时的 Agent 到点自动触发，触发参数（时刻、时区）来自该 Agent 自己的配置。
- **FR-006**: 系统 MUST 支持运行时登记/注销/查询一个 Agent；运行时路径与启动路径 MUST 走同一套校验、行为完全一致（同一异常类型、同一消息）。
- **FR-007**: 运行时登记一个带定时的 Agent 时，系统 MUST 为其定时任务保留一个可注销的句柄。
- **FR-008**: Agent 目录里的脚本 MUST 经系统已有的执行能力运行，且受约束：只放行白名单解释器、只能跑该 Agent 自己目录下的脚本。系统 MUST 明确"脚本能自行发起网络、绕过内置 HTTP 工具域名白名单"这一信任边界。
- **FR-009**: 主文件缺必填项（名字/运行配置）时，系统 MUST 加载报错并点名缺失项；配置引用系统未注册的能力时 MUST 记录告警；坏的 Agent MUST NOT 阻断系统启动或其余 Agent 的加载。

### Key Entities *(include if feature involves data)*

- **Agent 目录**: Agent 目录区里的一个目录，是"一个 Agent"的完整载体。含一份主文件（配置 + 正文）与可选的子指令/脚本/参考子资源。目录名即 Agent 名。
- **派生出的 Agent 配置**: 从主文件的配置部分派生出的、底座既有的 Agent 配置值对象——让 Agent 目录零改动复用整台底座。绑定了这个 Agent 的正文与资源目录位置。
- **注册表**: 已登记 Agent 配置的内存索引，支持启动批量登记与运行时增删查，两条来源共用同一套校验。
- **定时句柄**: 登记带定时的 Agent 时留下的可注销引用，为下一节运行时注销/更新铺路。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 往 Agent 目录区放一个目录、不写一行代码，扫描后它就出现在已注册 Agent 列表里。
- **SC-002**: 扫描一个放了 N 个合法 Agent 目录的目录区，注册表恰好出现 N 个 Agent、带定时的都进了定时器，无多余产物。
- **SC-003**: 一个带定时的 Agent 到达声明时间点会无人触发地跑完一轮、涉外调用都有审计记录；改其正文不重启，下一次触发即用新正文。
- **SC-004**: 一份非法 Agent 配置分别经启动路径与运行时路径加载，两边报出的错误类型与消息完全一致。
- **SC-005**: 一个带子指令与脚本的 Agent 运行时，正文在上下文、子指令在被读前不在上下文、脚本被执行且只有其产出（非代码）进上下文。
- **SC-006**: 自动化验收（课件 harness 全绿）覆盖以上关键回归点；`mvn clean verify` 全绿。

## Assumptions

- 前序节交付物均已就位并被直接复用：既有 Agent 配置注册表（16 节）、上下文加载器与 prompt 组装器（17 节）、定时器与其定时配置（25 节）、工具体系与统一工具抽象（17/20 节）、模型调用服务与审计（16 节）。
- 本节明确改造以下前序公共类型（当节课件已列为改造点）：Agent 配置注册表从不可变改为支持运行时增删（同一套校验）；定时器把注册循环体抽成单条注册方法并加可注销句柄表；上下文加载器从"按配置字段读技能文件全文注入"改为"注入 Agent 目录主文件正文"；Agent 配置来源从独立配置文件改为"扫描 Agent 目录派生"。
- 本节明确不做：Agent 版本管理、Agent 市场/共享、跨 Agent 的能力复用（每个 Agent 独立自足）、脚本的容器/网络隔离（只到解释器 + 目录白名单）、Agent 同名冲突策略、文件监听热加载（下一节做）、对外的动态管理接口（下一节做）。
- 无新增第三方依赖、无新增数据表；凭证走环境变量占位、不落明文。
- 开发在 `class-29` 分支进行，不另建分支。
