## User Scenarios & Testing *(mandatory)*

### User Story 1 - 与 Agent 多轮对话并得到最终回答 (Priority: P1)

运维/开发人员在终端启动 `oryxos chat`，用自然语言向 Agent 提问，Agent 结合上一轮上下文
连续多轮回答，直到用户退出。整个过程无需用户了解任何底层模型细节。

**Why this priority**: 这是「让一个 Agent 跑起来」的最小可用形态，也是所有后续能力（工具、
记忆、Web）的基座。没有它，其它一切无从演示。

**Independent Test**: 仅实现本故事即可独立验收——启动 `oryxos chat`，进行 ≥2 轮纯对话
（无工具），Agent 能记住同一会话内的上文并给出连贯回答。

**Acceptance Scenarios**:

1. **Given** 已配置好一个默认 Profile 且模型凭证有效，**When** 用户运行 `oryxos chat` 并
   输入一句问候，**Then** Agent 返回一段自然语言回答，会话保持打开等待下一轮输入。
2. **Given** 一轮对话已完成，**When** 用户追问一个依赖上文的问题（如"它多大了？"），
   **Then** Agent 的回答体现出对上一轮内容的理解。
3. **Given** 会话进行中，**When** 用户输入退出指令，**Then** 程序干净退出，无异常堆栈。

---

### User Story 2 - ReAct 循环支持工具调用调度机制 (Priority: P1)

当模型在某一轮响应中请求调用工具时，ReAct 循环能识别该请求、交由工具执行器执行、把结果
回填进对话历史，并让模型基于结果继续推理，直到产出最终回答。**本特性不交付任何具体工具**——
只实现并验证这条「检测 → 执行 → 回填 → 继续」的调度机制；真实内置/MCP 工具与「查天气穿衣」
端到端演示延后到工具特性/最后的演示阶段。因此本故事以**测试替身（fake 工具）**独立验证。

**Why this priority**: 工具调用调度是 ReAct 区别于普通聊天的本质环节，也是后续工具特性得以叠加
的接缝。机制必须在本特性就正确成型，故与 US1 同为 P1；但因不交付真实工具，验证走测试替身。

**Independent Test**: 注入一个返回固定数据的 fake 工具执行器，构造一段包含工具调用请求的模型
响应，验证循环执行了该工具、把结果回填、并据此继续推理直到最终回答；全程不接入任何真实工具。

**Acceptance Scenarios**:

1. **Given** 循环收到一个包含工具调用请求的模型响应且注入了 fake 工具执行器，**When** 循环处理该
   响应，**Then** 对应工具被执行恰好一次，结果被回填进对话历史，循环据此继续。
2. **Given** 一次提问需要连续两步工具调用（fake 执行器），**When** 循环运行，**Then** 依次完成
   多步调用并整合为一个最终回答，不把中间的工具结果当作最终答复返回给用户。
3. **Given** fake 工具执行器返回失败结果，**When** 循环收到该失败，**Then** 失败信息被回填进
   循环使模型可选择重试或说明，且单次失败不导致会话崩溃或无限重复。

---

### User Story 3 - 通过配置切换模型 Provider (Priority: P2)

维护者只修改 Profile 中的 Provider 配置（如从 A 模型换成 B 模型），无需改动任何代码，
即可让同一个 Agent 使用不同的大模型；多个不同 Provider 的 Agent 可在同一实例内并存。

**Why this priority**: 「不锁厂商、配置即 Agent」是 OryxOS 的核心卖点，但相对 US1/US2 属于
可后置的增强——先证明单 Provider 能跑通，再证明可切换。

**Independent Test**: 准备两个分别指向不同 Provider 的 Profile，分别对话，均能正常回答；
切换 Profile 后行为随配置变化，代码零改动。

**Acceptance Scenarios**:

1. **Given** 两个 Profile 指向不同 Provider，**When** 分别以各自 Profile 启动对话，
   **Then** 两者都能正常回答，且路由到各自正确的 Provider。
2. **Given** 一个 Profile 的 Provider 名称拼写错误或凭证缺失，**When** 启动，**Then** 系统
   给出清晰的错误提示，指明是哪个配置项的问题，而不是静默失败或含糊报错。

---

### Edge Cases

- 当一次提问触发的推理迭代达到上限仍未得出最终回答时，系统如何优雅收尾并告知用户？
- 当模型返回的工具调用请求指向一个不存在或未授权的工具时，如何处理？
- 当对话历史过长超出模型上下文窗口时，如何裁剪历史而不丢失关键上文？
- 当模型 API 超时、限流或返回错误时，用户看到的是什么？
- 当模型请求调用工具、但参数不合法（缺字段/类型错）时，如何反馈给模型或用户？
- 同一实例内并发多个会话时，各会话的上下文是否严格隔离、互不串扰？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个交互式命令行入口，支持在同一会话内进行多轮自然语言对话，
  并维护该会话的对话历史作为后续轮次的上下文。
- **FR-002**: 系统 MUST 允许 Agent 在单轮响应中判断是否需要调用工具；若需要，则由系统执行
  工具、将结果回填，并让 Agent 基于结果继续推理，如此循环直到产出最终回答。
- **FR-003**: 系统 MUST 为单次提问的推理循环设置可配置的迭代上限；达到上限仍无最终答案时，
  MUST 优雅终止并向用户返回可理解的说明，不得无限循环。
- **FR-004**: 每个工具在一次推理循环中被 Agent 请求调用时 MUST 恰好执行一次，MUST NOT 因
  框架行为被重复执行。
- **FR-005**: 系统 MUST 支持通过 Profile 配置选择模型 Provider，且切换 Provider 无需改动代码；
  多个指向不同 Provider 的 Agent MUST 能在同一实例内并存且路由正确。
- **FR-006**: 系统 MUST 以环境变量占位方式读取模型凭证等敏感配置，MUST NOT 要求用户把明文
  凭证写入 Profile。
- **FR-007**: 系统 MUST 在启动时校验 Profile 的必填项与合法性（如 Provider 名称、模型、凭证
  存在性），缺失或非法时给出明确指向具体配置项的错误，MUST NOT 静默失败。
- **FR-008**: 系统 MUST 记录每一次模型调用与每一次工具调用的审计信息（含耗时、成败），
  以便事后审计；审计记录的产生 MUST NOT 依赖于日志文本解析。
- **FR-009**: 当工具调用失败时，系统 MUST 把失败信息反馈进推理循环，使 Agent 能选择重试或
  向用户说明，MUST NOT 让单次工具失败导致整个会话崩溃。
- **FR-010**: 当对话历史超出模型可接受长度时，系统 MUST 按可配置的最近轮次上限裁剪历史，
  保留最近上下文。
- **FR-011**: 同一实例内的并发会话 MUST 各自隔离上下文，互不干扰。

### Key Entities *(include if feature involves data)*

- **会话 (Session)**：一次持续的多轮对话，包含按时间顺序排列的消息历史、所属 Agent（Profile）
  与状态（进行中/已结束）。
- **Profile**：一个 Agent 的完整定义，包含身份/人格、所选 Provider 与模型、可用工具集、
  以及迭代上限、历史轮次上限等运行参数。
- **消息 (Message)**：会话历史中的一条记录，可为用户输入、Agent 回答、工具调用请求或工具
  执行结果。
- **模型调用审计 (LLM Call)**：一次向模型发起调用的记录，含 Provider、模型、token 用量、耗时。
- **工具调用审计 (Tool Invocation)**：一次工具执行的记录，含工具名、入参、结果、成败、耗时。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户在配置好凭证后，能在 5 分钟内通过 `oryxos chat` 完成一次成功的多轮对话，
  无需查阅代码。
- **SC-002**: ReAct 循环的工具调用调度机制可用测试替身端到端验证——一次含工具调用请求的模型
  响应能触发"执行→回填→继续"直到最终回答。（真实工具的「查天气穿衣」端到端 Demo 延后。）
- **SC-003**: 切换 Provider 仅需修改配置、零代码改动，且请求能正确路由到目标 Provider（可用
  路由/配置级验证证明）。真实多 Provider 的运行时成功率对比留待多凭证就绪后进行。
- **SC-004**: 100% 的模型调用与工具调用都留下了可查询的审计记录（记录条数与实际发生次数一致）。
- **SC-005**: 任意单次提问的推理循环都在配置的迭代上限内终止，无一进入无限循环。
- **SC-006**: 配置错误（Provider 名/凭证缺失）时，用户能从错误信息中直接定位到出错的配置项，
  无需翻阅源码。

## Assumptions

- 首个交付渠道为 CLI（`oryxos chat`）；Web / 其它渠道在后续特性中处理。
- 本特性不交付任何具体工具（内置或 MCP）；只实现 ReAct 循环的工具调用调度机制，并以测试替身
  （fake 工具执行器）验证。完整工具体系（OryxTool / ToolRegistry / SandboxChecker / MCP）与
  「查天气穿衣」端到端演示是后续特性。循环需通过一个最小的工具执行器抽象来解耦，便于后续接入。
- 记忆能力（跨会话长期记忆）不在本特性范围内；本特性只维护单次运行内的会话上下文。
- 会话历史仅存内存，进程退出即丢失；跨重启持久化（SQLite）与长期记忆留待后续存储/记忆特性对接。
- 运行环境假定为可信内网，本特性不含认证、限流、RBAC。
- 模型 Provider 通过各自官方兼容接口访问，凭证经环境变量注入。
- 状态（会话历史）需可外置持久化，为后续跨重启恢复与分布式预留，但本特性可先以最小方式满足。
