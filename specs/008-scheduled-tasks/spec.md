# Feature Specification: 定时任务模块——让 Agent 到点自己干活

**Feature Branch**: `class-25`

**Created**: 2026-07-12

**Status**: Draft

**Input**: User description: "第25节需求：定时任务模块——让 Agent 不用人喂话、到点自己干活。定时任务是第三种触发源（CLI 人推、Web 人推、定时钟推），到点了系统自己拼一条消息喂给跟 CLI/Web 完全一样的统一编排入口，ReAct/Tool/Provider 一个字不用改。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 到点自动发起、无人参与 (Priority: P1)

运营方给某 Agent 在其配置里声明一条定时规则（cron 表达式 + 时区 + 到点要说的话）。进程启动后，到点时 Agent 自动发起一次对话、跑完整循环、结果按配置推出去，全程无人参与。这是"钟推"这个第三触发源存在的理由——很多场景（每日 PR 进度、日报、巡检告警）没人会天天手动去问。

**Why this priority**: 这是整个定时模块的核心价值：Agent 不用人喂话、到点自己干活。没有它，天气日报/科技日报这类"每日自动跑"的验收 Demo 根本立不起来。单独实现即可交付价值：一条 cron 配置就能让 Agent 自动运行。

**Independent Test**: 给一个 Agent 配一条"每分钟"定时规则，启动进程，到点观察 Agent 自动发起一次调用、走完循环、留下审计账（cron 链路本身只能真等一次）；单测层面直接调触发方法断言"消息交给了统一编排入口"。

**Acceptance Scenarios**:

1. **Given** 某 Agent 配了一条 cron+时区+消息的定时规则，**When** 进程启动完成注册、到达触发点，**Then** 系统自动拼出该消息交给统一编排入口，Agent 跑完整循环。
2. **Given** 一个 Agent 配了多条定时规则，**When** 启动，**Then** 每条都被独立注册、各按自己的 cron 到点触发。
3. **Given** 定时规则的 cron 表达式被改并重启（或重新加载配置），**When** 下一个触发点到达，**Then** 按新时间触发——无需改代码重新编译。

---

### User Story 2 - 重叠触发直接跳过、不并发跑两份 (Priority: P1)

一次循环跑得比调度间隔还久（例如每分钟触发但上一次还没跑完）。下一个触发点到达时，不能并发再跑一份，也不排队堆积——直接跳过本次。

**Why this priority**: 与 US1 同等关键。定时任务天然会遇到"跑得慢于间隔"的情况；若不防重叠，同一 Agent 会被并发触发多份，重复干活、重复推送、会话历史错乱。这是核心阶段必须守住的正确性。

**Independent Test**: 模拟某任务的锁已被占（上一次仍在跑），调用该任务的触发方法，断言这次没有把消息交给统一编排入口（本次被跳过）。

**Acceptance Scenarios**:

1. **Given** 某定时任务上一次触发仍在执行（其锁被占），**When** 下一个触发点到达，**Then** 本次触发直接跳过、不调用统一编排入口、不排队。
2. **Given** 某任务本次触发被跳过，**When** 上一次执行结束、再下一个触发点到达，**Then** 该次能正常触发（跳过是一次性的、不影响后续）。

---

### User Story 3 - 单次失败被隔离、留痕、不留死锁 (Priority: P1)

某次定时触发中途跑挂了。不能把整个调度器带崩，也不能悄悄失败什么都不留——得跟其他调用一样留痕；且不影响这个任务的下一次触发和其它任务；无论成败，占用的锁都要释放，不留死锁。

**Why this priority**: 定时任务无人值守，失败若把调度器带崩或悄悄吞掉，会让所有 Agent 的自动运行集体停摆且无人察觉。失败隔离 + 留痕 + 不死锁是无人值守场景的生命线。

**Independent Test**: 让统一编排入口在被调用时抛异常，调用触发方法断言：不外抛（调度器不死）；紧接着再触发一次能正常进入（"二进宫"证明锁在 finally 已释放、没永久卡死）。

**Acceptance Scenarios**:

1. **Given** 某次触发中统一编排入口抛异常，**When** 该次触发执行，**Then** 异常不外抛、调度器不崩、失败被记录（留痕）。
2. **Given** 上一次触发因异常失败，**When** 下一个触发点到达再次触发，**Then** 能正常进入执行（证明失败那次的锁已在 finally 释放）。
3. **Given** 某次触发失败，**When** 事后查审计，**Then** 这次失败的调用像人推触发的失败一样留在既有审计链路里（不为钟推单开审计逻辑）。

---

### Edge Cases

- **时区缺省**：定时规则未显式写时区时，如何取时区？（默认取服务器系统时区，见 Assumptions；但配置能显式覆盖是硬需求。）
- **非法 cron / 非法时区字符串**：某条规则的 cron 或时区非法时，MUST NOT 让整个注册过程崩溃拖垮其它合法规则的注册（单条坏规则隔离，记日志跳过）。
- **同一 Agent 配多条同 id 的规则**：任务标识重复时的锁归属——锁按任务标识维度分配，同标识共享一把锁。
- **进程退出**：定时线程 MUST NOT 阻止一次性命令（如交互对话）跑完后进程正常退出。
- **无任何定时规则**：所有 Agent 都没配 schedules 时，注册过程 MUST 正常空跑、不报错。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个调度层，启动时扫描所有 Agent 配置里声明的定时规则，逐条注册到系统的调度能力上。注册的触发规则 MUST 来自配置，MUST NOT 编译期写死（给 Agent 加/改定时不应要求改代码重新编译——"配置即 Agent"）。
- **FR-002**: 每条定时规则 MUST 支持 cron 表达式与显式时区；时区 MUST 能在配置里指定，MUST NOT 全由服务器系统时区替用户做主。
- **FR-003**: 到点触发时，本模块的唯一职责 MUST 是"拼一条消息交给统一编排入口"。消息内容由 Agent 配置/技能决定、交上去之后怎么处理由既有循环负责——这两件都 MUST NOT 由本模块承担（职责划窄，不得膨胀成工作流引擎）。
- **FR-004**: 系统 MUST 用按任务标识维度的进程内本地锁防重叠：上一次还没跑完、本次触发点到达时 MUST 直接跳过（不排队、不并行跑两份）。MUST NOT 引入分布式锁（核心阶段单实例）。
- **FR-005**: 单次触发失败 MUST 只记日志、不外抛、不拖垮调度器；失败的这次调用 MUST 仍走既有调用链路的完整审计（与人推触发的失败无区别，不为钟推单开审计逻辑）。无论成败，占用的锁 MUST 在 finally 释放，MUST NOT 留死锁。
- **FR-006**: 钟推也要落会话——会话身份 MUST 固定为三元组（channel=scheduler、user=scheduler、profile=该 Agent 名）。即同一 Agent 的历次定时触发 MUST 复用同一会话，对话历史自然累积、靠既有的最大历史轮数截断兜底，MUST NOT 为钟推新设任何会话概念。
- **FR-007**: 单条定时规则非法（cron/时区解析失败）MUST NOT 拖垮其它合法规则的注册与运行；坏规则记日志跳过。

### Key Entities *(include if feature involves data)*

- **定时规则（既有配置载体）**：一条定时任务的声明，含任务标识、cron 表达式、时区、到点要发给 Agent 的消息；已由前序节建全，本模块只消费。
- **调度层**：启动时把每条定时规则注册进系统调度能力；到点触发时按任务标识拿锁、拼消息、交给统一编排入口、失败记日志、finally 放锁。持有"任务标识→锁"的进程内表。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一条 cron+时区+消息的定时规则，到点即自动发起一次完整调用、无需任何人参与——由自动化 harness 断言"注册参数含配置的 cron 与时区""触发即交给统一编排入口"。
- **SC-002**: 上一次未跑完时，本次触发 100% 被跳过（0 并发第二份）——harness 恒绿。
- **SC-003**: 单次触发内部失败时，异常 0% 外抛、调度器不崩，且失败那次的锁 100% 被释放（下一次触发能正常进入）——关键回归，"二进宫"断言恒绿。
- **SC-004**: 同一 Agent 的两次定时触发拿到同一会话（三元组固定 scheduler/scheduler/profileName）——harness 断言。
- **SC-005**: 单条非法定时规则不影响其它规则——注册过程不整体崩溃。
- **SC-006**: 前序节全部测试保持全绿（本模块只消费既有入口、不改前序契约）。

## Assumptions

- 定时规则的配置载体（任务标识/cron/时区/消息）与 Agent 配置里的定时字段已由前序节建全（第16节全字段），本模块只消费、不改其结构或字面量。
- 统一编排入口（一次处理的编排者）、会话管理的"按三元组取或建会话"、Agent 配置的内存索引均前序节交付，本模块直接复用。
- 失败审计复用既有调用链路（人推触发失败怎么记账，钟推失败一样记账），本模块不新增审计表或审计逻辑。
- 时区未显式声明时默认取服务器系统时区；显式声明能覆盖（硬需求）。
- 调度机制使用框架自带的任务调度器 + cron 触发能力（不自造调度器）；核心阶段单实例，只解决"同一进程内不重叠"。
- 边界（本节明确不做）：多实例下"归哪个实例执行"的分布式协调（选主/分布式锁/租约）——扩展阶段；失败自动重试、失败 N 次自动告警——先不做，做到"失败不崩、留痕可查"即可；运行时经 API 增删改定时配置（免重启）——扩展阶段，核心阶段定时规则只写在 Agent 配置文件里、随进程启动注册、改配置要重启或重新加载才生效。
