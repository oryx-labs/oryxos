# 路线图

OryxOS 的开发理念是：**慢就是快，克制且聚焦**。先把单机的运行时内核做扎实，让一个节点上运行和管理一群 Agent 这件事真正可用、有人用，再在它之上逐步生长出分布式能力。分布式是终态愿景，但工程上坚持单机先行，每一步都走扎实，不一开始就堆一个无法落地的大型架构。

## 阶段一 — 单机运行时内核（当前）

在单节点上跑通五大核心能力：

- 一个目录 = 一个 Agent：一个 `AGENT.md` 目录，无需写代码——放进去就上线
- 多 Agent 并存：同一实例上运行多个独立 Agent
- 动态 Agent 生命周期：通过 REST 创建 / 更新 / 删除 / 调用，一句话生成草稿，上传即上线 —— *已完成*
- REST API：对外暴露所有能力，响应统一信封
- `/admin/` Web 管理台：Agent 管理、定时任务、会话、工具及运行时视图 —— *已完成*
- 动态 Provider 注册表：以 SQLite 为后端的 Provider CRUD，运行时换密钥 —— *已完成*
- 通知渠道：以 SQLite 为后端的 CRUD（飞书 / 企业微信 / 钉钉 / webhook） —— *已完成*
- 定时任务：按 Agent 的定时任务，含管理和执行历史 —— *已完成*
- 通过 REST 的工作区文件浏览 —— *已完成*
- MCP 集成：零代码接入任意社区 MCP server
- 全链路审计：`tool_invocations` 和 `llm_calls` 第一天起就写入

**目标**：把单节点运行和管理一群 Agent 做到真正可用，而不是停留在原型阶段。

## 阶段二 — 底座分布式（规划中）

让运行时可以水平扩展：

- 实例无状态化：OryxOS 进程内不保留本地状态
- 状态外置：Session、记忆、审计数据存储在实例之外
- 多副本部署：多个 OryxOS 实例运行在负载均衡后面
- 高可用：消除单点故障

## 阶段三 — 跨节点 Agent 协作（愿景）

让不同节点上的 Agent 能够相互发现、委托和协同：

- 对接 A2A（Agent-to-Agent）开放协议
- 跨节点 Agent 发现与注册
- Agent 之间可靠的异步任务委托
- 跨节点消息投递保障
